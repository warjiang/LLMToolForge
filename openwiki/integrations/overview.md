# Integrations: MCP, Connectors, and Skills

LLMToolForge has three major integration surfaces that extend the agent's capabilities: **MCP servers** (tool/resource providers), **Connectors** (third-party API actions), and **Skills** (prompt + tool augmentations synced to external agents).

## MCP Servers

MCP (Model Context Protocol) servers provide tools, resources, and prompts to agents. LLMToolForge lets you manage servers, inspect their capabilities, and bridge their tools into the agent runtime.

**UI**: [`/src/pages/mcp/McpPage.tsx`](/src/pages/mcp/McpPage.tsx), [`/src/pages/mcp/McpDialog.tsx`](/src/pages/mcp/McpDialog.tsx)  
**Rust backend**: [`/src-tauri/src/mcp/mod.rs`](/src-tauri/src/mcp/mod.rs)  
**Tool bridge**: [`/src/lib/agent/tools/mcp.ts`](/src/lib/agent/tools/mcp.ts)

### Transport Support

| Transport | Description |
|---|---|
| **stdio** | Launch a local process and communicate via stdin/stdout. PATH completion for `npx`, `uvx` via login-shell PATH recovery. |
| **SSE** | Server-Sent Events over HTTP for streaming communication. |
| **HTTP** | Standard HTTP transport for request/response communication. |

### Server Management

- **CRUD**: Create, edit, delete MCP server configurations with dynamic forms based on transport type
- **Import**: Bulk import from standard `mcpServers` JSON format (duplicate names are skipped with notification)
- **Built-in servers**: A curated set of MCP servers available for one-click install with install/uninstall lifecycle gating (PR #51-53 series)
- **Enable/disable**: Toggle individual servers on and off

### MCP Inspector

The built-in Inspector lets you interact with MCP servers directly:

1. **Connect**: Performs the `initialize` handshake with the server
2. **Browse Tools**: Lists all available tools with their JSON Schema definitions
3. **Call Tools**: Execute tools with parameter forms auto-generated from JSON Schema, or raw JSON input
4. **Read Resources**: Browse and read server resources
5. **Get Prompts**: Retrieve server prompts

The Inspector is implemented in Rust ([`/src-tauri/src/mcp/mod.rs`](/src-tauri/src/mcp/mod.rs)) and invoked via Tauri commands:
- `mcp_connect` — Initialize connection
- `mcp_list_tools` — Discover tools
- `mcp_call_tool` — Execute a tool
- `mcp_read_resource` — Read a resource
- `mcp_get_prompts` — Get prompts

Results are displayed in the UI in real-time. Error details from server load failures are surfaced to the user (PR #59).

### Agent Tool Bridge

When an MCP server is enabled and an `AgentDefinition` has MCP tools enabled, the server's tools are wrapped as [agent tools](/openwiki/agent/overview.md). The agent can call them through the standard tool loop: the Pi runtime or AAP host tool bridge calls `mcp_call_tool`, which invokes the Rust MCP client.

## Connectors

Connectors provide access to **1,000+ third-party platforms** (10,000+ actions) via the [OpenConnector](https://github.com/oomol-lab/open-connector) runtime, which runs as a managed sidecar binary.

**UI**: [`/src/pages/connectors/ConnectorsPage.tsx`](/src/pages/connectors/ConnectorsPage.tsx)  
**Rust supervisor**: [`/src-tauri/src/connector/mod.rs`](/src-tauri/src/connector/mod.rs)  
**Frontend API**: [`/src/lib/connector/api.ts`](/src/lib/connector/api.ts)  
**Sidecar**: [`/sidecar/connector/`](/sidecar/connector/)

### Architecture

The connector runs as a **separate binary** (Bun-compiled TypeScript) with its own SQLite database. The Rust supervisor manages its lifecycle:

```
Agent → connector_list_apps / connector_search_actions / …
   → Frontend → Rust connector supervisor → OpenConnector sidecar (port 4160)
```

### Credential Isolation

This is a critical security design choice:

- Third-party credentials (API keys, OAuth2 tokens) are stored **exclusively** in the connector's own SQLite database (`<app-data>/connector/`)
- Credentials are **never written** to the app's data store or included in storage sync
- The admin token used for frontend-to-connector communication is generated by Rust and persisted separately

### Sidecar Lifecycle

| Command | Purpose |
|---|---|
| `connector_start` | Start the OpenConnector binary on a random/configurable port (binds `127.0.0.1` only) |
| `connector_stop` | Stop the sidecar |
| `connector_status` | Check if running, get port |
| `connector_open_console` | Open the built-in web console in the system browser |

The sidecar can also be controlled from the **system tray** (PR #61).

### Connector UI Features

- **Runtime status card**: Start/stop, port display, web console link
- **Provider catalog**: Browse and search 1,000+ providers
- **Credential management**: Per-provider API Key forms and OAuth2 client configuration with system-browser authorization flow
- **Action search and debug**: Find actions, auto-generate parameter forms from JSON Schema, execute and view results
- **Run history**: Track past action executions

### Agent Tools

Connectors expose **4 discovery tools** to agents (defined in [`/src/lib/agent/tools/connector.ts`](/src/lib/agent/tools/connector.ts)):

| Tool | Purpose |
|---|---|
| `connector_list_apps` | List available third-party apps/providers |
| `connector_search_actions` | Search for actions across providers |
| `connector_get_action_guide` | Get the parameter schema and guide for a specific action |
| `connector_execute_action` | Execute an action with parameters |

This discovery-based approach avoids flattening 10,000+ actions into the agent's context. The tools are toggled per `AgentDefinition` or per session (default: off). If the connector isn't running, tools return a descriptive error guiding the user.

## Skills

Skills are reusable prompt + tool augmentations that can be synced to external agents (Claude Code, Codex, Cursor, etc.).

**UI**: [`/src/pages/skills/SkillsPage.tsx`](/src/pages/skills/SkillsPage.tsx)  
**Marketplace**: [`/src/lib/skillMarket/`](/src/lib/skillMarket/)  
**Sync**: [`/src/lib/skillSync.ts`](/src/lib/skillSync.ts), [`/src/lib/skillTargets.ts`](/src/lib/skillTargets.ts)  
**Agent integration**: [`/src/lib/agent/tools/skills.ts`](/src/lib/agent/tools/skills.ts)

### Skill Lifecycle

1. **Create**: Manual skills with name, description, content, tags
2. **Install from market**: Search on [skills.sh](https://skills.sh) or enter a GitHub `owner/repo` to browse available skills
3. **Multi-file support**: Skills can include `SKILL.md` plus `references/`, scripts, and binary resources (UTF-8 text or base64-encoded)
4. **Update detection**: Installed skills track their source (`owner/repo`, ref, path) and content hash for detecting upstream updates
5. **Sync to targets**: Copy or symlink skill files to external agent directories (Claude Code, Codex, Cursor, etc.)

### Marketplace

Two market providers:
- **GitHub**: Enter `owner/repo`, app fetches the repo tree, lists available skill directories
- **skills.sh**: Search the skills.sh registry; content is still pulled from the respective GitHub repos

Optional GitHub token configuration increases API rate limits.

### Skill Targets

Skills can be synced to 11 supported agents:

`claude_code` | `codex` | `cursor` | `opencode` | `gemini_cli` | `github_copilot` | `kiro` | `qoder` | `droid` | `openclaw` | `hermes`

Sync modes:
- **Copy**: Full file copy to the target directory
- **Symlink**: Symbolic link (for local development workflows)

The sync engine is in Rust (`sync_skills_to_targets` command in [`/src-tauri/src/lib.rs`](/src-tauri/src/lib.rs)), with path sanitization and traversal guards.

### Skill Requirements

Skills can declare external binary dependencies in their `SKILL.md` frontmatter (`metadata.requires.bins`). The app checks whether these binaries are available on PATH using the `check_skill_bins` Tauri command and surfaces missing dependencies to the user.

### Agent Skill Integration

In the Pi agent runtime, enabled skills are injected into the system prompt via the `<available_skills>` block (Pi-style). The `load_skill` tool lets the agent fetch a skill's full content on demand. Skills are configured per `AgentDefinition`.

### Write-Connector Skill

The repository includes a built-in skill for developing new OpenWiki connectors at [`/skills/write-connector/`](/skills/write-connector/).
